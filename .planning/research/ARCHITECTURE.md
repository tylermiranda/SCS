# Architecture Patterns

**Domain:** Web Scraping and Visualization Tool (Static JSON Architecture)
**Researched:** 2024-05-24
**Overall confidence:** HIGH

## Recommended Architecture

A decoupled "Data Pipeline to Static Dashboard" architecture. This approach splits the system into two distinct environments: a server-side execution environment for scraping, and a static client-side environment for visualization, bridged by a static JSON contract.

### Component Boundaries

| Component | Responsibility | Communicates With |
|-----------|---------------|-------------------|
| **Scraper** (`scraper/`) | Navigates the target ASP.NET site using Playwright, extracts raw HTML, and parses it into structured data. | Target Website (HTTP), Data Transformer (In-memory) |
| **Data Transformer** | Cleans, formats, and aggregates the raw scraped data into the final JSON schema required by the dashboard. | Scraper, File System |
| **Data Storage** | Acts as the handoff point. A static `data.json` file. | Data Transformer (Writer), Dashboard Client (Reader) |
| **Dashboard Client** (`dashboard/`) | Vanilla JS SPA built with Vite. Fetches the static JSON, initializes Chart.js, and handles user interactions (sorting, filtering). | Data Storage (HTTP GET via browser), User |

### Data Flow

The system operates on a unidirectional data flow (Pipeline pattern):

1. **Extraction:** A scheduled cron job or manual trigger executes the Node.js scraper.
2. **Ingestion:** Playwright interacts with the ASP.NET site and extracts raw data.
3. **Transformation:** The script transforms the raw data into a predefined JSON schema.
4. **Handoff:** The script writes the JSON payload directly into the dashboard's `public/` directory (e.g., `dashboard/public/data.json`).
5. **Consumption:** A user visits the static dashboard. On load, the client-side JavaScript performs a `fetch('/data.json')` or similar.
6. **Presentation:** The dashboard parses the JSON and renders the data into charts and tables.

### Suggested Build Order (Dependencies)

To ensure smooth development, components should be built in the order of data flow, starting from the data contract.

1. **Data Contract Definition:** Define the exact structure of `data.json`. The dashboard depends on this shape, and the scraper targets this shape.
2. **Scraper Extraction:** Build the Playwright script to successfully navigate the ASP.NET site and log raw data. *(Depends on target site structure)*
3. **Data Transformation & Handoff:** Format the raw data to match the Data Contract and write it to the `public/` directory. *(Depends on Scraper Extraction)*
4. **Dashboard Skeleton & Mock Data:** Build the UI layout and setup Vite. Use a mock `data.json` matching the contract to begin wiring up Chart.js. *(Can happen in parallel with steps 2 & 3)*
5. **Dashboard Integration:** Replace mock data with actual data generated by the scraper. Implement sorting, filtering, and search functionalities. *(Depends on Data Transformation & Handoff)*

## Patterns to Follow

### Pattern 1: Strict JSON Schema Contract
**What:** The structure of the JSON file must be strictly defined and documented.
**When:** Always, as this is the only interface between the scraper and the dashboard.
**Example:**
```json
{
  "lastUpdated": "2023-10-27T10:00:00Z",
  "properties": [
    {
      "address": "123 Main St",
      "appraisalValue": 250000,
      "year": 2023
    }
  ]
}
```

### Pattern 2: Idempotent Scraping
**What:** The scraper should completely overwrite the JSON payload (or merge cleanly) without creating duplicate entries if run multiple times.
**When:** During the handoff phase to ensure the dashboard always has a clean, accurate dataset.

## Anti-Patterns to Avoid

### Anti-Pattern 1: Scraping on Demand
**What:** Triggering the Playwright scraper from a user action in the dashboard.
**Why bad:** ASP.NET scraping via Playwright is slow and prone to timeouts or blocking. It breaks the static hosting model and requires a live backend server.
**Instead:** Run the scraper on a background schedule (cron) and serve the pre-computed JSON to users instantly.

### Anti-Pattern 2: Tight Coupling of Scraper and UI
**What:** Having the scraper directly manipulate the dashboard's DOM or importing UI libraries into the scraping script.
**Why bad:** Makes testing difficult and breaks the boundary between the data generation and presentation layers.
**Instead:** Strictly use the JSON file as the only communication channel.

## Scalability Considerations

| Concern | At 1 neighborhood (Current) | At 10 neighborhoods | At Entire County (100k+ properties) |
|---------|-----------------------------|---------------------|-------------------------------------|
| **Scraper Runtime** | Minutes (Single thread Playwright) | Hours (Requires batching/concurrency) | Days (Needs distributed scraping, proxies, and queueing) |
| **JSON Payload Size** | < 1 MB (Loads instantly) | 5-10 MB (Needs gzip/brotli compression) | 100+ MB (Unusable. Must abandon static JSON for an API/DB or implement pagination/chunked JSON files) |
