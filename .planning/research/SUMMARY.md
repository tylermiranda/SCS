# Project Research Summary

**Project:** Sedgwick County Tax Scraper
**Domain:** Web Scraping and Visualization Tool (Static JSON Architecture)
**Researched:** 2024-05-24
**Confidence:** HIGH

## Executive Summary

The project is a Web Scraping and Visualization Tool designed to extract property tax data from the Sedgwick County ASP.NET site and present it in a lightweight dashboard. Experts build this type of product using a decoupled "Data Pipeline to Static Dashboard" architecture, where a backend scraper runs on a schedule (e.g., via GitHub Actions) to generate a static JSON file that a frontend dashboard consumes.

The recommended approach is to use Node.js and Playwright for resilient data extraction from the complex ASP.NET site, explicitly avoiding reverse-engineering the server state in favor of genuine browser interactions. The frontend should be a simple Vanilla JS/HTML SPA built with Vite and utilizing Chart.js for visualization, adhering strictly to a lightweight constraint. This architecture eliminates the need for live databases or backend servers, greatly simplifying hosting and maintenance through static providers like GitHub Pages.

Key risks include scraper failure due to ASP.NET ViewState desync and frontend performance degradation from a bloated static JSON payload. These are mitigated by using Playwright's auto-waiting locators for natural navigation, and by preprocessing data into columnar formats or using downsampling and Chart.js decimation techniques on the client side.

### Architecture Approach

The architecture follows a unidirectional "Data Pipeline to Static Dashboard" flow, maintaining a strict static JSON contract between the backend and frontend.

**Major components:**
1. **Scraper (`scraper/`)** — Navigates the target ASP.NET site using Playwright, extracts raw HTML, and parses it into structured data.
2. **Data Transformer** — Cleans, formats, and aggregates the raw scraped data into the final JSON schema required by the dashboard.
3. **Data Storage** — Acts as the handoff point via a static `data.json` file.
4. **Dashboard Client (`dashboard/`)** — Vanilla JS SPA built with Vite that fetches the static JSON and handles UI rendering and user interactions.

### Critical Pitfalls

1. **ASP.NET ViewState & Postback Desync** — Do not attempt to reverse-engineer API calls; use genuine Playwright interactions (`locator.click()`, `locator.selectOption()`) to handle hidden ViewState fields naturally.
2. **Static JSON Payload Bloat** — Use columnar JSON structures, pre-process/downsample data, and ensure Gzip/Brotli compression to prevent browser crashes.
3. **Chart.js Rendering Jank on Large Datasets** — Disable animations, disable individual point rendering, and use the Decimation plugin for >5000 data points.
4. **Flaky Scraping via Brittle Selectors & Hard Sleeps** — Strictly use auto-waiting user-facing locators (`getByRole`), never use hard sleeps, and wait for explicit data container state changes.

## Implications for Roadmap

Based on research, suggested phase structure:

### Phase 1: Scraper Core Implementation
**Rationale:** Data extraction is the foundation; the data contract must be established before the frontend can be fully developed.
**Delivers:** Playwright extraction scripts targeting the ASP.NET site.
**Addresses:** Resilient HTML Parsing, Configurable Targeting, Clear CLI Output.
**Avoids:** ASP.NET ViewState & Postback Desync, Flaky Scraping via Brittle Selectors.

### Phase 2: Data Pipeline & Handoff
**Rationale:** Raw data needs formatting into the strict static JSON contract to decouple the backend from the frontend.
**Delivers:** Data transformer script, strict JSON schema definition, and static JSON file export logic.
**Uses:** Node.js, Static JSON.
**Implements:** Data Transformer, Data Storage Component.

### Phase 3: Dashboard UI / Visualization
**Rationale:** Once the JSON contract and a sample payload exist, the static frontend can be reliably built.
**Delivers:** Vanilla JS/Vite SPA displaying time-series charts, KPI cards, and data tables.
**Uses:** Vite, Vanilla JS/HTML, Chart.js.
**Implements:** Dashboard Client Component.
**Avoids:** Chart.js Rendering Jank on Large Datasets.

### Phase 4: Interactivity & Automation
**Rationale:** The final polish step adds user interaction features and automates the scraping pipeline for a hands-off operational state.
**Delivers:** Sorting, filtering, search functionality, and GitHub Actions cron workflows.
**Addresses:** Multi-column Sorting, Global Text Search & Filtering, Scheduled Execution (Cron).
**Avoids:** Silent Failures in Cron Execution.

### Phase Ordering Rationale

- **Dependencies:** The frontend dashboard directly depends on the data contract generated by the scraper pipeline. Therefore, building the scraper and transformer first ensures the dashboard has real (or accurately mocked) data to consume.
- **Architecture Grouping:** Phase 1 and 2 focus entirely on the backend `scraper/` component and data handoff, while Phase 3 and 4 focus on the `dashboard/` component and overall system orchestration.
- **Pitfall Avoidance:** By front-loading ASP.NET scraping in Phase 1, the highest-risk technical hurdle is tackled early before investing time in UI polish.

### Research Flags

Phases likely needing deeper research during planning:
- **Phase 1:** Handling specific ASP.NET structures of Sedgwick County, identifying specific locators and potential anti-bot measures.
- **Phase 4:** Specific GitHub Actions runner limitations regarding Playwright browser installation and execution times.

Phases with standard patterns (skip research-phase):
- **Phase 2:** Standard Node.js data normalization and JSON writing are straightforward.
- **Phase 3:** Vite and Chart.js integration are well-documented and standard.
